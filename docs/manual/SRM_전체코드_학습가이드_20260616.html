Part 1 — 시스템 전체 구조

STM32 MCU가 지상반(TML UART)에서 작업을 받고, 상태머신으로 주행·승강·포크를 제어하며, EtherCAT으로 인버터를 돌리고, 0x30 폴링으로 상태를 돌려준다.

main 루프와 Manager 체계

// main.c while(1) — 매 사이클 반복
AlarmManager();      // DI/DO, 에러, Save_* 트리거
FpgaManager();       // FPGA 통신
McuComManager();     // MCU 보조 통신
TMLComManager();     // ★ 지상반 UART 수신/송신
EtherNetManager();   // 이더넷
ECAT_Manager();      // ★ 인버터 PDO 갱신
SRM_Manager();       // ★ SRM 제어 진입
  └─ SRM_Machine_Process()      // SSR/연결 등
  └─ SRM_Machine_Run_Process()  // ★★ 상태머신 본체
  └─ HMI_ModbusTCP_Proc()
임베디드 관점: super loop 구조. RTOS 태스크가 아니라 while(1) 협력적 멀티태스킹. 상태머신은 SRM_Machine_Run_Process() 한 함수의 giant switch. 1ms 타이머 ISR에서 m_pgmEnv.timer1ms++ → 모든 딜레이의 시계.

데이터 3층 — 반드시 구분

[입력층]  0x41 WorkCmdSTR  →  rxCmdOrder()  →  Store_Work_Data*()  →  m_WorkData[s_WorkInx]
                                                                              │
[실행층]  m_pgmEnv.SRM_RunMode (enumSRM_RUN_SEQ)  ←── SRM_Machine_Run_Process() switch
          m_pgmEnv.SRM_RunTimer, Sub_Run_Step (축별 서브시퀀스)
                                                                              │
[출력층]  Save_ForkWork_*() / Save_ForkMove_*()  →  m_St (StatusSTR)  →  0x30 응답
          save_error_code()  →  m_St 에러 + m_BKSram.AlarmLog
초보가 가장 많이 헷갈리는 것: m_WorkData(실행용)와 m_St(보고용)를 동시에 본다.
디버깅 시 "지상반 화면에 뭐가 보이나" → m_St / "MCU가 실제 뭘 실행하나" → m_WorkData + RunMode

rxCmdOrder() 진입 조건 (거부 사유):

  • Fault 상태, Manual SW, RunMode≠STAND_BY, 포크 센터 이탈, Auto 모드 아님
  • GET/PUT 시 StartOn==0 이면 NON_ONLINE
  • 화물 센서 이상 (GoodsFlag, DISABLE 설정 시 생략)

수락 후 Store_Work_Data / Store_Work_Data_1 / Store_Work_Data_Fix 등 OrderCode·ForkAct에 따라 m_WorkData 채움.

0x50 Start 흐름

// com_tml.c rxCmdStart() 요지
m_pgmEnv.SRM_RunMode = RUN_SEQ_START;
// 이후 dev_SRM.c SRM_Machine_Run_Process()의 case RUN_SEQ_START: 에서
// m_WorkData[s_WorkInx].ForkAct 에 따라 LOADING/UNLOADING/MOVE 분기

0x30 상태 보고

지상반이 주기 폴링 → MCU가 m_St 내용을 패킷으로 응답.

  • m_St.ForkWork[i] — 작업별 OrderStatus, OrderProcess(스텝)
  • m_St.Inv_St[4] — 축별 위치·속도·GoodsFlag
  • m_St.SRM_Status1/2 — Auto, StartOn, Fault 비트

Save_* 함수가 m_St를 갱신. UART로 즉시 push가 아니라 다음 0x30 폴링 때 반영.

Part 3 — RunMode (상태머신)

개념

m_pgmEnv.SRM_RunMode = enum enumSRM_RUN_SEQ (dev_SRM.h, 73개)

SRM_Machine_Run_Process() = 거대한 switch-case. 매 main 루프마다 현재 case 한 스텝만 실행.

case 안에서 조건 충족 시 m_pgmEnv.SRM_RunMode = 다음값 으로 전이. break 필수.

명명 함정 8선

GoodsFlag_Or_MoveHome
함정: 이름은 '화물 OR 홈이동' 같지만 축마다 의미 다름
실제: 포크=화물유무, 주행/승강=홈복귀중 플래그. 반드시 축(INV_*) 맥락에서 읽을 것.
Rack_fork_obstruction
함정: 랙-포크 물리적 간섭 센서 같음
실제: WCS가 '이미 InPosition'이라 스킵하려 할 때 MCU가 '그래도 이동해야 함' 표시. 1이면 InPosition이어도 재이동.
ENABLE_TRAV_LEFT_MOVE_BRAKE_RELEASE
함정: 좌측 이동 시에만 브레이크 해제
실제: 컴파일 옵션(0/1). 주행+승강 이동 전 브레이크 해제 확인 서브시퀀스 on/off. LEFT는 레거시 네이밍.
Save_* vs Store_*
함정: 둘 다 '저장' 같음
실제: Save=m_St 지상반 보고, Store=m_WorkData 내부 큐.
SRM_TaskMode / 0x40
함정: RunMode와 같은 것
실제: 0x40 Task Order 전용. 일반 0x41 작업과 별도 경로.
RETURN_HOME vs ORIGIN
함정: 둘 다 홈
실제: ORIGIN=축 원점 설정(기준점 잡기), RETURN_HOME=운전 중 홈 위치로 복귀.
TRAV_MOVE_FORWARD / REVERSE
함정: 좌/우 방향
실제: 엔코더 좌표 기준 TargetPos > StartPos면 FORWARD. 현장 좌/우는 파라미터·설치 방향에 따름.
m_pgmEnv vs m_St
함정: 둘 다 상태
실제: m_pgmEnv=MCU 내부( RunMode, 타이머, 통신링크). m_St=지상반에 보여주는 얼굴.

브레이크 해제 타임아웃 3초는 어디서?

정답: LOADING_COMPLETE → RUN_SEQ_COMPLETE → (PLC_COM) → RUN_SEQ_STAND_BY

주행 방향은 누가 결정?

정답: Check_Trav_Move_Dir() — StartPos vs TargetPos → TRAV_MOVE_FORWARD/REVERSE

에러 80-1 (WORK_TIMEOUT) 서브코드 nFlag=0x01이면?

정답: 주행(INV_TRAVEL) 브레이크만 해제 실패. 승강은 OK.

m_WorkData는 extern인가?

정답: 아님. dev_SRM.c static. 다른 파일에서 직접 접근 불가.

  1. 이 가이드 4주 로드맵 완주
  2. dev_SRM.c 에 한 개 case를 골라 조건·전이·Save 호출 손으로 주석 달기
  3. 연습: 딜레이 200ms 추가 — Read_DelayTime / DEF_DELAY_* / case 타이머 중 어디를 바꿀지 찾기만 (실제 수정은 테스트 환경에서)
  4. 연습: 새 RunMode case 하나 설계해보기 (종이에 전이도 그리기)
  5. 심볼 사전으로 모르는 이름 즉시 검색 — 이 습관이 자립의 핵심
